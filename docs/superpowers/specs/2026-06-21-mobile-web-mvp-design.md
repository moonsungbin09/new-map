# 모바일 웹 MVP 1차 배포 설계서

## 1. 목표
- 오늘 1차 배포 목표는 모바일 브라우저에서 다음 3가지를 동작시키는 것이다.
  1. 새 장소 등록
  2. 장소 목록 조회
  3. 장소 상세 조회
- 로그인은 2차 보완으로 미룬다.

## 2. 범위
### 포함
- 모바일 웹 화면 (반응형)
- 장소 등록/목록/상세 API
- Azure SQL 저장
- Azure App Service 배포

### 제외 (2차)
- 회원가입/로그인
- 장소 수정/삭제
- 고급 공유 기능
- 운영 모니터링 고도화

## 3. 아키텍처
- 클라이언트: 모바일 웹 (정적 파일 기반 UI)
- 서버: REST API (장소 등록/조회)
- 데이터베이스: Azure SQL
- 배포: Azure App Service 단일 서비스에 API와 정적 파일을 함께 배포 (1차 단순화)

## 4. 주요 컴포넌트
### 4.1 프론트엔드
- 화면: 홈(목록), 장소 등록, 장소 상세
- 핵심 동작:
  - 장소명/좌표/유형 입력 후 저장
  - 목록에서 장소 확인
  - 항목 선택 시 상세 확인
- 입력 검증:
  - 장소명 필수
  - 위도/경도 범위 체크
  - 잘못된 값은 저장 차단 및 안내 문구 표시

### 4.2 백엔드 API
- `POST /places`: 장소 등록
- `GET /places`: 목록 조회
- `GET /places/{id}`: 상세 조회
- 응답 형식은 JSON으로 고정한다.

### 4.3 데이터베이스
- 저장 테이블은 MVP 기준 `Place` 중심으로 구성한다.
- 핵심 컬럼:
  - `id`
  - `name`
  - `latitude`
  - `longitude`
  - `level_type`
  - `description`
  - `created_at`

## 5. 데이터 흐름
1. 사용자가 모바일 웹에서 장소 정보를 입력한다.
2. 프론트엔드가 API로 등록 요청을 보낸다.
3. API가 유효성 검증 후 Azure SQL에 저장한다.
4. 저장 결과를 JSON으로 응답한다.
5. 프론트엔드는 결과를 받아 목록/상세 화면에 반영한다.

## 6. 에러 처리 정책
- 네트워크 실패: "네트워크를 확인한 뒤 다시 시도해 주세요."
- 서버/DB 실패: "잠시 후 다시 시도해 주세요."
- 잘못된 입력값: 필드별로 즉시 안내 문구를 표시한다.
- 앱이 멈추는 형태의 실패 대신, 사용자 안내와 재시도 경로를 우선 제공한다.

## 7. 테스트 기준 (오늘 1차)
- 모바일에서 접속 가능
- 장소 등록 성공
- 목록/상세 조회 성공
- 잘못된 입력 차단 확인
- 네트워크 끊김 시 오류 안내 표시

## 8. 배포 기준
- Azure App Service에서 API와 정적 파일 모두 정상 응답
- Azure SQL 연결 정상
- 실제 모바일 브라우저에서 등록/조회 흐름 1회 이상 성공

## 9. 2차 보완 백로그
- 로그인/권한 도입
- 수정/삭제 기능
- 공유 기능 고도화
- 모니터링/로그 체계 강화
